Multi-Tenant Intune Deployment

Designing and implementing Microsoft Intune device management across multiple MSP client tenants — 1,000+ Windows endpoints spanning hybrid Azure AD and cloud-only environments.

Microsoft IntuneEntra ID Windows AutopilotConditional Access PowerShellWin32 Apps

Overview

Managing endpoints across multiple enterprise clients from a single MSP introduces complexity that single-tenant deployments don't face — each client has its own Entra ID tenant, its own compliance requirements, its own application portfolio, and its own mix of hybrid and cloud-only devices. A misconfigured policy in one tenant cannot affect another, and the baseline needs to be consistent enough to maintain at scale while flexible enough to accommodate per-client needs.

This deployment is currently in the design and testing phase, using the MSP's own tenant as a sandbox before rollout to client environments. All enrollment scenarios, compliance policies, and application deployments are validated end-to-end before touching any client.

Enrollment Strategy

Windows Autopilot ● CONFIGURED

Zero-touch provisioning for new and reimaged devices. Hardware hash uploaded at procurement. OOBE bypassed — device enrolls and configures automatically on first boot. Enrollment Status Page blocks user access until required apps and policies are applied.

Auto-Enrollment ● CONFIGURED

Automatic MDM enrollment triggered by Entra ID device group membership. No manual enrollment steps required. Group assignment drives both enrollment and subsequent policy and application targeting.

Application Deployment

Line-of-business applications are packaged as Win32 (.intunewin) packages using the Microsoft Win32 Content Prep Tool. Each app requires a silent install command, uninstall command, detection rule, and return code configuration before deployment.

Win32 App packaging workflow
# 1. Package the installer
IntuneWinAppUtil.exe -c .\source -s setup.exe -o .\output

# 2. Upload to Intune with:
#    - Silent install:   setup.exe /S /norestart
#    - Silent uninstall: setup.exe /uninstall /S
#    - Detection rule:   File exists at C:\Program Files\App\app.exe
#    - Return codes:     0=success, 1707=success, 3010=reboot required

# 3. Assign to device group (Required vs Available)
# 4. Monitor deployment status in Intune portal

Per-Client Configuration

On top of the baseline, each client gets a configuration layer scoped to their specific device groups — preventing any cross-tenant policy bleed.

OneDrive KFM

Known Folder Move silently redirects Desktop, Documents, and Pictures to OneDrive on first sign-in. No user interaction required. Configured via Intune device configuration profile.

SharePoint Sync

Client SharePoint site URLs configured for automatic sync. Silent configuration — sites appear in File Explorer without the user needing to navigate to SharePoint.

Mapped Drives

Network drive mappings deployed via PowerShell scripts through Intune. Scoped to client-specific device groups. Handles authentication differences between hybrid and cloud-only identity models.

Known Gotchas

Hybrid join timing

Devices in hybrid environments can take up to 24 hours to reflect Entra ID join status after on-prem domain join. Premature Intune enrollment attempts before sync completes result in "Not Applicable" deployment status — device appears enrolled but policies don't apply.

ESP timeout

Applications with long install times can cause the Enrollment Status Page to time out. Timeout value must be tuned based on the largest required app deployment in each client's profile.

Win32 detection rules

Inaccurate detection rules cause Intune to repeatedly attempt reinstallation. Every app's detection rule must be validated against a clean install before broad deployment.

Cloud-only mapped drives

Traditional UNC path mapped drives require line-of-sight to on-prem file servers. Cloud-only devices need SharePoint/OneDrive as the replacement — mapped drive scripts must detect device type and handle accordingly.
