Centralized Log Management

Fleet-wide syslog collection across 34 LXC containers deployed via Ansible, organized into security streams, with real-time dashboards in Graylog.

GraylogAnsible OpenSearchrsyslog MongoDBProxmox

Overview

Running 34 LXC containers across a self-hosted homelab generates a lot of log data — auth events, service starts and stops, kernel messages, application errors. Without centralized collection, investigating anything means SSH'ing into individual containers and grepping through logs manually. This project solves that by shipping every container's logs to a single Graylog instance, organized into streams, and searchable from one interface.

The architecture is deliberately simple: rsyslog (already installed by default on Debian/Ubuntu) forwards logs over UDP to Graylog. No agents to install, no complex pipelines. The deployment was automated via Ansible across all 34 hosts in a single playbook run.

Ansible Deployment

Rather than configuring rsyslog manually on each container, the forwarding config was deployed fleet-wide in a single Ansible playbook run from a dedicated control node (LXC 140).

SSH keys were bootstrapped across all 34 hosts via Proxmox pct exec — which works regardless of whether SSH was pre-configured on each container, making it suitable for bootstrapping a mixed fleet from scratch.

playbooks/rsyslog-graylog.yml
- name: Configure rsyslog forwarding to Graylog
  hosts: lxc_containers
  become: true
  tasks:
    - name: Ensure rsyslog is installed
      apt:
        name: rsyslog
        state: present

    - name: Deploy Graylog forwarding config
      copy:
        dest: /etc/rsyslog.d/10-graylog.conf
        content: |
          *.* @192.168.x.x:1514;RSYSLOG_SyslogProtocol23Format

    - name: Restart rsyslog
      service:
        name: rsyslog
        state: restarted

Log Streams

Raw syslog from 34 containers would be one noisy feed. Graylog streams route messages into buckets based on field-matching rules, making it practical to search and alert on specific categories.

Security Events ● ACTIVE
facility_num = 4 (auth)
facility_num = 10 (authpriv)
message contains "Failed password"
message contains "sudo"
message contains "authentication failure"

Captures SSH activity, PAM auth events, sudo usage, and login failures across the entire fleet.

System Events ● ACTIVE
facility_num = 0 (kern)
message contains "Failed to start"
message contains "Out of memory"
message contains "Started" / "Stopped"

Kernel messages, service lifecycle events, OOM kills, and systemd unit failures.

Application Logs ● ACTIVE
facility_num = 3 (daemon)
facility_num = 16–17 (local0/1)

General daemon and application-level syslog output from all services.

Troubleshooting: MongoDB FTDC Segfault

Graylog's MongoDB component crashed on startup with a SIGSEGV on every attempt, despite the host CPU (Xeon E5-2630 v3) correctly reporting AVX support. After ruling out OS mismatch, memory pressure, file ownership issues, and container privilege levels, the root cause was isolated via strace and a timeout-based test:

MongoDB 8.x's FTDC (Full Time Diagnostic Data Capture) thread uses TCMalloc's periodic CPU profiling sampler, which crashes on this specific Xeon microcode revision even though raw AVX execution works correctly. The crash happened consistently at ~30 seconds after startup — exactly when FTDC's first collection interval fires.

/etc/systemd/system/mongod.service.d/override.conf
[Service]
ExecStart=
ExecStart=/usr/bin/mongod --config /etc/mongod.conf \
  --setParameter diagnosticDataCollectionEnabled=false

Disabling FTDC via systemd drop-in resolved the crash. MongoDB has been stable since.
